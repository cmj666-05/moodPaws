# MoodPaws 页面开发文档（按当前代码进度更新）

## 1. 文档目标

本文档以当前仓库实际实现为准，用于同步 `MoodPaws` 现阶段页面结构、视觉风格、数据接入方式与开发完成度，作为后续页面补齐、UI 细化、Android 打包和设备数据联调的参考。

当前项目已落地 4 个底部 Tab 页面入口：

1. 项圈
2. 宠舍
3. 情绪
4. 社交

当前 4 个入口现已按文件语义完成调整：**宠舍入口对应 `DashboardView.vue`，社交入口对应 `SocialView.vue`；社交入口当前收敛为开发中过渡页。**

## 4. 当前通用开发约束

### 4.1 页面组织方式

当前项目采用以下方式组织页面：

- 一级页面集中在 `src/views/`
- `App.vue` 负责底部 Tab 切换
- 各页面内部自己管理图表、地图、展示卡片与轮询逻辑
- 当前没有抽出统一详情页路由

### 4.2 数据接入方式

当前已实现页面的数据主链路，实际上是：

```text
后端 API（moodpaws-server）
→ usePetApi()
→ 页面响应式状态
→ 页面组件渲染
```

当前 `usePetApi()` 已对以下数据进行拉取：

- `/telemetry/latest`
- `/telemetry/metrics/HeartRate/history?limit=30`
- `/telemetry/location/track?limit=200`
- `/emotion/latest`

其中情绪主展示当前由 `/emotion/latest` 提供；当设备上报 `PetHouse:Mood` 时，后端会优先将其映射为中文 `currentMood` 返回给情绪页与项圈页。

说明：

- 当前页面层**主要消费后端 API**，不是页面直接订阅 MQTT。
- MQTT 相关逻辑当前主要保留在 `moodpaws-server/` 与联通性测试脚本中，前端页面已不再保留 MQTT 直连代码。
- 页面数据通过轮询自动刷新。
- **当前宠舍页仍使用本地假数据首版；社交过渡页会复用 `usePetApi()` 的调试信息能力。**

### 4.3 当前公共状态

现阶段页面中已实际出现的通用状态包括：

- 加载中
- 加载失败
- 等待数据
- 在线
- 连接中
- 离线
- 等待定位
- 地图加载失败

旧文档中提到的“无数据、网络断开、预警已读”等状态，当前未形成统一组件体系。

## 12. 下一步开发建议（按当前实现衔接）

建议后续按以下顺序推进：

1. 为当前 4 个一级页面补充统一的状态组件与空态组件
2. 为项圈页优先补详情页或模块弹层
3. 将宠舍页从“假数据业务页”逐步接入真实 API 数据
