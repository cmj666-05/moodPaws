# MoodPaws 页面开发文档

## 1. 文档目标

本文档用于明确 `MoodPaws` 安卓 App 的页面结构、子界面拆分、核心模块、交互状态与开发顺序，作为后续产品设计、前端开发、安卓打包与 MQTT 数据联调的统一依据。

当前产品围绕 4 个核心业务界面展开：

1. 宠物项圈数据界面
2. 寄养屋数据界面
3. 宠物情感分析界面
4. 宠物 NFC 社交界面

文档默认基于以下前提：

- 平台为 `Web 浏览器` 和 `Android App`（双平台交付）
- 当前项目技术基座为 `Vite + Vue 3 + Capacitor`
- 数据来源以 `MQTT + IoT 设备上报` 为主
- 高德地图在 Web 端和 Android 端均可加载
- MQTT 连接在 Android 端正常，Web 端可能受阿里云跨域限制
- 第一阶段优先完成页面框架、展示模型、状态流转和假数据占位
- 第二阶段再逐步接入真实设备数据与历史记录

## 3. 统一设计与开发规范

### 3.1 视觉风格

- 设计关键词：`科技感`、`温暖陪伴`、`轻数据仪表盘`
- 色彩倾向：暖白底、橙色强调、青绿色辅助、蓝色 AI 信息强调
- 页面结构：顶部摘要区 + 中部卡片内容区 + 底部导航
- 卡片风格：大圆角、弱阴影、清晰分区、图表轻量

### 3.2 页面布局约定

- 移动端基准尺寸：`360 x 800dp`
- 页面顶部保留宠物/房间摘要卡
- 中部采用滚动卡片布局
- 每页至少提供 1 个主指标概览区和 2 个次级详情区
- 所有实时数据模块都要提供更新时间与在线状态

### 3.3 通用状态设计

每个页面和模块都应至少支持以下状态：

1. 首次加载
2. 数据加载中
3. 正常展示
4. 无数据
5. 数据异常
6. 设备离线
7. 网络断开

### 3.4 通用交互规则

- 点击卡片进入详情页
- 图表支持时间范围切换，例如 `今日`、`7天`、`30天`
- 预警消息支持查看详情和标记已读
- 实时数据默认自动刷新
- 用户主动下拉刷新时应重新拉取或重新订阅

## 4. 页面一：宠物项圈数据界面

用于展示佩戴智能项圈的宠物实时健康、活动状态、定位轨迹和情绪识别结果，是最核心的设备监测页面。

- 直观看到运动量和活动轨迹

## 12. 数据接入建议

为了避免后续页面和设备数据强耦合，页面层不要直接消费原始 MQTT 消息，建议统一通过页面 ViewModel 暴露数据。

建议结构：

```text
MQTT 原始消息
→ 解析层 parser
→ 领域模型 model
→ 页面视图模型 viewModel
→ 页面组件
```

这样可以避免以下问题：

- 页面直接依赖设备字段命名
- 后期字段变化时需要大面积改 UI
- 历史数据和实时数据无法统一

## 13. 当前结论

当前项目后续页面开发应围绕以下原则展开：

1. 先做 4 个核心一级页面骨架
2. 每个一级页面至少拆出 4 到 6 个子界面
3. 先完成展示型页面，再接入控制型能力
4. 先假数据跑通，再逐步接入真实 MQTT 数据
5. 所有页面统一走组件化与数据模型映射层
