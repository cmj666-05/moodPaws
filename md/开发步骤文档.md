# MoodPaws 开发步骤文档

## 1. 当前项目判断

当前仓库不是已经完成过业务重构的项目，而是一个刚创建好的 `Vite + Vue 3 + Capacitor` 默认应用。

现状确认：

- `src/App.vue` 仍然是默认示例页面。
- 目前没有 MQTT 配置层、页面路由规划、状态层、接口层。
- 文档 `md/task.md` 和 `md/walkthrough.md` 记录的是另一个已重构项目的目标结构，里面出现了 `pages.json`、`pages/sub/...` 等 `uni-app` 风格目录，不能直接照搬到当前项目。
- 因此本项目应按“从 0 到 1”重新落地，只把那几份文档里的业务目标和阿里云接入信息作为需求来源。

## 2. 第一优先级：先验证能否拿到阿里云 IoT 数据

这一步必须先做，否则后续界面开发都只能用假数据。

### 2.0 连接身份修正说明

注意：物理设备和前端不能共用同一个阿里云 IoT 身份 ID。

- `DHT11` 只作为真实硬件设备使用，负责上报温湿度等传感器数据。
- 前端 Web/App 必须改用独立的虚拟设备身份 `webapp` 建立连接，不能再直接拿 `DHT11` 的三元组登录。
- 前端当前的“连接设备”应理解为“登录身份切换到 webapp，监听的数据仍然来自 DHT11 的上报 Topic”。

前端连接应统一使用以下虚拟身份：

```json
{
  "ProductKey": "k1wxakcs6OI",
  "DeviceName": "webapp",
  "DeviceSecret": "03a0edc6d46dd71a4eecd81ec95f5bd0"
}
```

### 2.1 已整理出的连接信息

- Broker Host: `iot-06z00b1eo2alugk.mqtt.iothub.aliyuncs.com`
- 设备 ProductKey: `k1wxakcs6OI`
- 硬件设备名: `DHT11`
- Web 端设备名: `webapp`
- 设备上报源 Topic: `/sys/k1wxakcs6OI/DHT11/thing/event/property/post`
- 前端订阅目标 Topic: `/k1wxakcs6OI/webapp/user/get`

补充约束：

- “连接身份” 和 “被监听设备” 是两件事，不要再混用。
- 前端连接参数里的 `DeviceName` 固定使用 `webapp`。
- 业务展示的数据来源仍然是 `DHT11` 的属性上报消息。
- 前端不再直接订阅 `DHT11` 的系统 Topic，而是订阅云产品流转转发后的自定义 Topic。
- 不能退回去复用 `DHT11` 身份给前端直连。

### 2.1.1 云产品流转解析配置

当前阿里云云产品流转已经提供了从设备消息到前端消息的转发逻辑：

```javascript
var data = payload("json");

writeIotTopic(1001, "/k1wxakcs6OI/webapp/user/get", data);
```

这段配置的含义是：

- 先把进入规则引擎的消息体按 JSON 解析为 `data`。
- 再把 `data` 写入自定义 Topic `/k1wxakcs6OI/webapp/user/get`。
- 前端 `webapp` 应订阅这个自定义 Topic 获取转发后的设备数据。
- `DHT11` 的系统 Topic 继续作为源数据入口，但不再作为前端直连订阅目标。

### 2.2 已添加的测试脚本

我已经补了一个最小测试脚本：

- [scripts/test-aliyun-mqtt.mjs](/D:/desktop/moodPaws/scripts/test-aliyun-mqtt.mjs)

并在 [package.json](/D:/desktop/moodPaws/package.json) 中添加了命令：

```bash
npm run test:aliyun -- device-tcp
npm run test:aliyun -- web-wss
```

两种模式含义：

- `device-tcp`：使用 `md/a.md` 中现成的 DHT11 设备凭证，通过 `1883` 直接连 MQTT。这个最适合先判断“当前网络能不能直达阿里云、凭证是否有效、设备主题上是否有数据”。
- `web-wss`：使用 `md/ALIYUN_IOT_CONFIG.md` 中的 `webapp` 三元组，通过 `wss://...:443/mqtt` 连接。这个更接近后续 Vite/Capacitor 前端的接入方式。

注意：

- `device-tcp` 订阅源 Topic：`/sys/k1wxakcs6OI/DHT11/thing/event/property/post`
- `web-wss` 应订阅前端自定义 Topic：`/k1wxakcs6OI/webapp/user/get`

### 2.3 连通性测试的执行顺序

执行原则：

- 每一次连接测试都必须由你手动执行，我只提供命令和结果判读规则，不再代替执行。
- 如果需要我协助排查，我只基于你贴出的命令输出继续分析。

1. 先关闭 Clash Verge，避免阿里云 MQTT 域名或 `1883/443` 连接被代理干扰。
2. 先跑 `device-tcp`，确认能否订阅并收到 DHT11 的属性上报。
3. 如果 `device-tcp` 成功，再跑 `web-wss`，确认前端是否能从自定义 Topic 收到流转后的消息。
4. 两个都成功后，前端开发默认接真实 MQTT。
5. 如果都失败，先不要写页面，先查网络、Topic、设备在线状态和阿里云控制台权限。

### 2.4 手动执行命令

请在项目根目录 `D:\desktop\moodPaws` 下手动执行以下命令：

```powershell
npm.cmd run test:aliyun -- device-tcp
npm.cmd run test:aliyun -- web-wss
```

说明：

- `device-tcp`：用于验证 `DHT11` 设备凭证、目标 Topic 和设备上报链路。
- `web-wss`：用于验证前端 `webapp` 虚拟身份是否能通过 WSS 方式连接阿里云。
- 在 PowerShell 中优先使用 `npm.cmd`，避免 `npm.ps1` 被执行策略拦截。

### 2.5 测试通过标准

通过标准不是“能连上”，而是“能在目标 Topic 上收到真实消息”。至少要拿到一条 DHT11 上报载荷，才能判定数据链路可用。

在当前方案下，通过标准应拆成两层：

- 设备链路通过：`device-tcp` 能在 `/sys/k1wxakcs6OI/DHT11/thing/event/property/post` 收到真实消息。
- 前端链路通过：`web-wss` 能在 `/k1wxakcs6OI/webapp/user/get` 收到规则引擎转发后的真实消息。

## 4. 实际开发步骤

## 阶段 A：完成阿里云数据验证

1. 执行 `device-tcp` 测试，记录是否收到消息。
2. 执行 `web-wss` 测试，记录是否收到消息。
3. 把实际收到的 MQTT 载荷保存为样例 JSON，作为前端联调基准。
4. 如果暂时拿不到真数据，先整理一份 mock 数据，但不要跳过真实链路验证。

输出物：

- 一份真实 MQTT 样例消息
- 一份连通性结论：成功 / 失败 / 失败原因

## 阶段 B：建立应用骨架

1. 删除默认示例组件 `HelloWorld.vue` 的使用。
2. 引入 Vue Router，建立 5 个一级视图：总览、项圈、寄养、情感、社交。
3. 搭建统一布局：顶部标题区、内容区、底部导航。
4. 定义主题变量、间距系统、卡片样式和响应式断点。
5. 保留适合 Capacitor 的移动端布局，不先做复杂桌面端交互。

输出物：

- 可切换的多页面骨架
- 不再包含默认模板内容的首页

## 阶段 C：封装 MQTT 基础设施

1. 新建 `src/services/mqtt/config.js`，集中放 Broker、Topic、模式选择。
2. 新建 `src/services/mqtt/client.js`，封装连接、重连、订阅、取消订阅、错误处理。
3. 新建 `src/composables/useMqtt.js`，向页面暴露响应式连接状态和最新消息。
4. 增加消息解析层，把阿里云物模型消息转换为前端需要的结构。
5. 区分“连接状态错误”和“业务数据为空”，避免页面误判。

输出物：

- 页面可复用的 MQTT 客户端层
- 一套统一的数据转换函数

## 阶段 D：先做总览页，再做分页面

开发顺序不要平均铺开，应按“先总览、后细分”推进。

1. 先做总览页：
   - 展示 MQTT 连接状态
   - 展示最近一次温湿度/设备上报时间
   - 展示到 4 个功能页的入口
2. 再做项圈页：
   - 当前文档中提到心率、血氧、体重、GPS、健康预警
   - 如果阿里云当前并没有这些字段，先明确标记“待设备侧接入”，不要伪装成真实数据
3. 再做寄养屋页：
   - 优先接 DHT11 温湿度
   - 交互控制项只有在确认存在下行 Topic 时才开发
4. 再做情感分析页：
   - 如果没有 AI 情绪判定数据源，先做占位和数据接入口，不做假算法
5. 最后做社交页：
   - NFC 与当前 MQTT 文档关系最弱，放最后

输出物：

- 一个能看真实 IoT 数据的总览页
- 若干逐步补齐的业务页

## 阶段 E：Capacitor 与 Android 联调

1. 先保证 `npm run build` 正常。
2. 再同步到 Android 工程。
3. 验证 Android WebView 是否能正常使用 MQTT over WSS。
4. 检查移动端后台切换、断网重连、页面恢复后的 MQTT 状态。
5. NFC 功能仅在确定 Android 原生能力接入方案后再展开。

输出物：

- Android 可运行包
- 一份真机联调记录

## 5. 明确哪些内容现在不要先做

当前不建议优先做：

- 大规模页面视觉精修
- 情感分析假算法
- 没有真实 Topic 支撑的控制面板
- 没有 Android 原生接入方案的 NFC 业务

原因很简单：项目还没验证最基础的数据链路，先把“真实数据进页面”做通才是关键路径。

## 6. 建议的最近执行顺序

按这个顺序推进最稳：

1. 关闭 Clash 后执行阿里云连通性测试。
2. 保存真实 MQTT 消息样例。
3. 清理默认 Vite 页面，建立路由和底部导航。
4. 封装 MQTT 服务层。
5. 做总览页并接入真实数据。
6. 逐页补业务页面。
7. 最后再做 Android 真机联调。

## 7. 当前结论

基于现在的仓库状态，可以明确判断：

- 这个项目还没有进入“重构收尾”阶段，而是“刚建好默认应用”的起步阶段。
- 原文档里的完成清单不能视为当前项目已完成内容。
- 当前第一步不是写页面，而是先验证你现在这套阿里云 MQTT 凭证和网络是否真的能拿到数据。

在你关闭 Clash 后，就可以直接执行本地连通性测试。

### 8.2 `device-tcp` 测试结果

执行命令：

```bash
npm.cmd run test:aliyun -- device-tcp
```

结果：

- 已成功连接到 `mqtt://iot-06z00b1eo2alugk.mqtt.iothub.aliyuncs.com:1883`
- 已成功订阅 `/sys/k1wxakcs6OI/DHT11/thing/event/property/post`
- 命令输出明确出现 `MQTT connected.` 和 `Subscribed. Waiting for a message...`

判断：

- `DHT11` 设备凭证可用
- 该 Topic 的订阅动作可用
- 当前缺的不是连接能力，而是“测试窗口内没有设备上报消息”

### 8.3 当前结论修正

现在可以确认以下事实：

- 阿里云 MQTT 服务地址可达
- `DHT11` 设备凭证可正常连接并订阅其系统 Topic
- `webapp` 虚拟身份也可正常连接并订阅自定义 Topic `/k1wxakcs6OI/webapp/user/get`
- 当前两条测试命令都已通过“连接层”和“订阅层”验证
- 60 秒超时只能说明测试窗口内没有命中消息，不能再据此判断链路未通
