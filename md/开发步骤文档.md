  - `PetHouse:CH2O`
  - `PetHouse:VOC`
  - `PetHouse:MQ135`
  - `PetHouse:Weight`

## 5. 当前正确的开发顺序

这个项目现在不应该直接进入完整业务开发，而应该按下面顺序推进。

### 阶段 A：确认测试页稳定收消息

1. 在浏览器打开当前测试页。
2. 使用 `petInfo` 身份建立 WSS 连接。
3. 确认页面能稳定收到 `/k1wxaEnEO8L/petInfo/user/get` 的真实消息。
4. 确认页面展示值和原始 JSON 一致。

通过标准：

- 页面能看到 `Collar` 作为来源设备。
- 至少能看到一组真实环境指标。
- 页面刷新后可再次连接。

### 阶段 B：抽离 MQTT 基础设施

当前测试页是“能用”的，但还不是正式结构。下一步应抽离：

1. `src/services/mqtt/config.js`
2. `src/services/mqtt/client.js`
3. `src/composables/useMqtt.js`
4. `src/utils/pet-house-parser.js`

目的：

- 不要把连接参数、订阅逻辑、消息解析都堆在 `App.vue`
- 后续页面复用时才能保持清晰

### 阶段 C：先做总览页

总览页第一版只做真实数据展示，不做复杂业务。

建议第一版包含：

- MQTT 连接状态
- 当前来源设备名
- `PetHouse` 设备卡片组
- `Collar` 设备卡片组
- 最后更新时间

当前已落地到首页项圈页的内容：

- 心率卡片
- 血氧卡片
- XYZ 三轴运动数据
- GPS 地图区域
- 经纬度显示
- 实时轨迹线
- 轨迹点数量
- Android 运行时地图加载限制

这一页完成后，项目才算真正脱离“测试状态”。

### 阶段 D：再拆业务页面

当总览页稳定后，再拆成业务模块页面：

1. `PetHouse` 环境监测页
2. `Collar` 姿态页
3. `Collar` 健康页
4. `Collar` 定位页
5. 历史数据页

注意：

- 没有真实 Topic 支撑的控制功能，也不要现在先做

## 5.1 后续开发必须遵守的数据策略

这一条很关键，后面开发都按这个规则走：

1. 不能直接把 MQTT 原始 `items` 结构塞进页面。
2. 必须先经过解析层，把真实字段映射成页面模型。
3. 同一条消息只更新它携带的字段，没有出现的字段保留上次有效值。
4. 只有用户主动清空或重新进入页面时，才允许清空界面缓存。
5. 原始 JSON 仍然要保留一份，方便安卓真机联调排错。
6. 页面顶部显示的是“最后一条消息来源设备”，不代表整页所有数据都来自同一个设备。
7. 总览页和后续页面都应按设备归属组织数据，不要再回退到抽象分类优先。

## 5.2 后续开发建议补齐的内容

后续真正会需要继续开发的内容如下：

1. 把 MQTT 配置改成环境变量，不继续把设备密钥直接写在源码里。
2. 为总览页增加“最后一次成功收包时间”和“数据超时告警”。
3. 为定位数据增加地图展示或坐标展示组件。
4. 为心率血氧增加异常阈值标记。
5. 为历史数据页补时间序列缓存结构，不要到时候再返工解析层。
6. 为安卓 App 增加断网重连、切后台恢复和启动自动连接策略。
7. 为 `PetHouse` 和 `Collar` 建立独立的数据模型和历史缓存结构。

### 阶段 E：最后做 Android 联调

1. 确保 `npm run build` 正常
2. 同步 Capacitor Android
3. 检查 Android WebView 下 MQTT over WSS 是否稳定
4. 检查断网重连、切后台重连

## 7. 当前结论

现在可以明确下结论：

- 当前前端测试身份应使用 `petInfo`，而不是之前的 `webapp`
- 当前前端订阅 Topic 应使用 `/k1wxaEnEO8L/petInfo/user/get`
- 当前页面应围绕 `PetHouse` 和 `Collar` 两个真实设备的字段展示
- 当前项目第一阶段已经不再是“验证字段猜测”，而是“把双设备真实消息接成正式页面结构”
- 当前地图实现以 `Android App` 交付为准，不再单独面向 Web 产品做兼容设计
- 当前项目已经具备签名 keystore、固定 SHA1 和正式 APK 输出能力

所以接下来的开发重心，应该是从“测试页”过渡到“正式总览页”。
