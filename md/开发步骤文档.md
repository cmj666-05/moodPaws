## 1. 当前项目现状

当前仓库是基于 `Vite + Vue 3 + Capacitor` 的跨平台应用，同时支持 **Web 浏览器** 和 **Android App** 两种运行环境。

当前已确认：

  - `宠舍` -> `src/views/social/SocialView.vue`
  - `情绪` -> `src/views/emotion/EmotionView.vue`

- 当前正式 MQTT 数据源恢复为历史上已验证可工作的聚合 Topic：`/k1wxaEnEO8L/petInfo/user/get`。
- `petInfo` 作为虚拟接收设备负责接收转发后的多设备消息，真实来源设备通过 payload 中的 `deviceName` 区分。
- 高德地图在 Web 端和 Android 端均可加载。
- 旧的前端 MQTT 直连链路（`src/composables/useMqtt.js`、`src/services/mqtt/*`）已经从仓库删除，页面主流程只保留 `usePetApi -> moodpaws-server`。

- 当前宠舍页已回归正确入口，但仍以本地假数据首版为主。
- 当前社交页已收敛为建设中过渡页，并保留 telemetry 调试信息用于联调。

## 5. 当前正确的开发顺序

这个项目现在不应该直接进入完整业务开发，而应该按下面顺序推进。

### 阶段 A：确认基础链路稳定

1. 启动 `moodpaws-server`
2. 访问 `/api/health` 确认 MQTT 已连接并已订阅
3. 确认 telemetry 接口能返回稳定数据
4. 确认前端页面展示值和接口返回一致

通过标准：

- 页面能看到 `Collar` 作为来源设备。
- 至少能看到一组真实环境指标。
- 页面刷新后可再次连接。

### 阶段 B：保持 API 主链路稳定

当前正式结构的重点已经不是把 MQTT 逻辑继续塞回前端，而是保持：

- `moodpaws-server`
- `src/composables/usePetApi.js`
- 页面展示层

这条链路稳定。

### 阶段 C：先完善现有一级页面

建议优先完善这 4 个一级页面：

1. `项圈`：继续补详情页与异常态
2. `宠舍`：逐步把本地假数据替换成真实 API 数据
3. `情绪`：补充更细的分析详情页
4. `社交`：从过渡页推进到真实 NFC 社交流程

### 阶段 D：再拆业务详情页

当一级页稳定后，再拆成业务模块页面：

1. `PetHouse` 环境监测页
2. `Collar` 姿态页
3. `Collar` 健康页
4. `Collar` 定位页
5. 历史数据页
6. 社交记录与好友详情页

注意：

- 没有真实 Topic 或 API 支撑的控制功能，不要现在先做。

## 5.1 后续开发必须遵守的数据策略

这一条很关键，后面开发都按这个规则走：

1. 不能直接把 MQTT 原始 `items` 结构塞进页面。
2. 必须先经过解析层，把真实字段映射成页面模型。
3. 同一条消息只更新它携带的字段，没有出现的字段保留上次有效值。
4. 只有用户主动清空或重新进入页面时，才允许清空界面缓存。
5. 原始 JSON 仍然要保留一份，方便联调排错。
6. 页面顶部显示的是"最后一条消息来源设备"，不代表整页所有数据都来自同一个设备。
7. 总览页和后续页面都应按设备归属组织数据，不要再回退到抽象分类优先。
8. 宠舍页与社交页的页面语义要与入口保持一致，不再混用。

## 5.2 后续开发建议补齐的内容

后续真正会需要继续开发的内容如下：

1. 把 MQTT 配置改成环境变量，不继续把设备密钥直接写在源码里。
2. 为总览页增加"最后一次成功收包时间"和"数据超时告警"。
3. 为心率血氧增加异常阈值标记。
4. 为历史数据页补时间序列缓存结构，不要到时候再返工解析层。
5. 为安卓 App 增加断网重连、切后台恢复和启动自动连接策略。
6. 为 `PetHouse` 和 `Collar` 建立独立的数据模型和历史缓存结构。
7. 为宠舍页补真实视频、控制、历史趋势与告警能力。
8. 为社交页补 NFC、好友、匹配和隐私设置能力。

### 阶段 E：Android 联调与构建

1. 确保 `pnpm build` 正常
2. 同步 Capacitor Android
3. 检查 Android 端页面与地图能力是否正常
4. 检查断网重连、切后台恢复

## 8. 当前结论

现在可以明确下结论：

- 当前用于 MQTT 联通性验证的测试身份应使用 `petInfo`，而不是之前的 `webapp`
- 当前测试订阅 Topic 应使用 `/k1wxaEnEO8L/petInfo/user/get`
- 当前页面应围绕 `PetHouse` 和 `Collar` 两个真实设备的字段展示
- 当前项目第一阶段已经不再是“验证字段猜测”，而是“把双设备真实消息接成正式页面结构”
- 当前页面正式链路已经稳定为 `usePetApi -> moodpaws-server`，Web 端和 Android 端都可用于页面联调
- 真实 MQTT 消息接入、订阅与解析能力集中在 `moodpaws-server` 与测试脚本侧，而不在前端页面侧
- 运动数据区域已从原始 X/Y/Z 三轴数据改为步数估算展示
- 当前项目已经具备签名 keystore、固定 SHA1 和正式 APK 输出能力
- 当前宠舍 / 社交入口语义已经校正，后续应分别沿各自方向推进

所以接下来的开发重心，应该是从“已有一级页雏形”继续推进到“各业务页完整化”。
